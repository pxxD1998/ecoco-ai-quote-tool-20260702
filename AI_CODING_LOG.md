### 獨立二審發現與修正

第一次 Spec Review 回傳 `REQUEST_CHANGES`（P0 0、P1 3），其中兩項是產品缺陷：

1. 新版歷史資料刪空後，程式又從舊版 `quoteHistory` 匯入，使已刪除資料復活。
2. 單價欄空白時，`Number('')` 會轉成 `0`，導致標示必填的欄位仍可儲存。

第三項 P1 是審查期間候選仍在變動，無法把 PASS 綁定到唯一版本。處理方式是先完成修正與文件，再建立含雜湊的不可變候選快照重新審查。

兩個缺陷的針對性 RED：

```text
PASS (0) FAIL (2)
```

修正後的針對性 GREEN：

```text
PASS (2) FAIL (0)
```

修正規則：

- 只要新版 storage key 存在，即使值是空陣列，也不再重新讀取舊 key；首次遷移後移除舊 key。
- 數字輸入空白保留為空白狀態，不先轉成 0；驗證明確區分未填與負數／格式錯誤。

### 第二輪 Risk Review

固定快照的風險審查又用隔離 Playwright 案例找出五項風險：

1. 匯入 JSON 可將字串放入數字欄位，經 HTML attribute template 造成可執行內容。
2. 畸形品項可能先寫入 localStorage，再於渲染時失敗，留下持久污染資料。
3. 同編號匯入會靜默取代既有報價。
4. TWD 小數折扣在各欄位分別四捨五入後無法對帳。
5. 流水號到 1000 後，原本只辨識三位數的規則會再次產生 1000。

五個案例納入正式測試後，修正前：

```text
PASS (0) FAIL (5)
```

修正後：

```text
PASS (5) FAIL (0)
```

主要修正：

- 匯入前驗證完整 schema 與資料型別，任何一筆畸形就整批拒絕，通過後才寫入 localStorage。
- 既有報價優先；同編號匯入明確略過，不靜默覆蓋。
- 所有插入 HTML attribute 的值仍進行跳脫，形成第二層防護。
- TWD 僅接受整數金額；USD 最多兩位小數；稅額依幣別最小單位四捨五入。
- 流水號辨識三位以上數字，999 之後為 1000、1001。

## 8. 改善版關鍵決策

### 決策 A：維持純前端

沒有因為要「像正式產品」就直接新增登入、後端與雲端資料庫。原因是這些功能會引入：

- 客戶資料外傳風險。
- 帳號與權限模型。
- 憑證管理。
- 服務維運與部署成本。

對目前作品目標而言，JSON 可攜備份是較小、較穩定且可驗證的解法。

### 決策 B：先補可信度，不堆功能

優先順序是：

1. 金額不能因負數而錯誤。
2. 報價編號不能重複。
3. 重灌後資料能手動還原。
4. 自動測試能重現與防止回歸。
5. 商務欄位及文案必須誠實。

### 決策 C：品牌降級為背景脈絡

首頁明確標示：

- 求職作品展示。
- 源自 ECOCO 實地測驗。
- 非 ECOCO 官方產品。

避免作品展示被誤解為公司授權或正式系統。
