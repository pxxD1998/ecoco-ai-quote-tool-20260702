# AI Coding 對話紀錄

## 背景

實測題目要求：將每天約 30 分鐘的手工報價單，整理成 Web 工具。重點不是功能越多越好，而是能判斷在 1 小時內交付哪些可用功能，並保留 AI coding 對話紀錄。

## 我給 AI 的任務方向

我要求 AI 協助建立一個可在 GitHub Pages 部署的靜態 Web 工具，條件如下：

1. 必須在 15:30 前可交付。
2. 不使用後端與資料庫，降低部署失敗風險。
3. 優先完成題目列出的 13 個可用功能。
4. 介面使用繁體中文。
5. 交付物必須包含部署 URL、Source Code、AI coding 對話紀錄。

## AI 協助拆解出的實作方案

AI 建議用純 HTML / CSS / JavaScript 完成，原因是：

- 不需要安裝框架，速度最快。
- GitHub Pages 可直接部署靜態檔案。
- 使用者與主管打開網址即可測試。
- PDF 可先用瀏覽器列印另存 PDF，避免引入 PDF 套件造成時間風險。
- 歷史報價可先用 localStorage，避免後端與帳號系統。

## 實作過程摘要

1. 建立 `index.html`：包含基本資料、產品清單、報價單預覽、歷史報價與功能對照。
2. 建立 `style.css`：完成 RWD 版面、品牌色、報價單列印樣式。
3. 建立 `script.js`：完成新增 / 刪除產品、自動計算、稅率、折扣、多幣別、localStorage 歷史報價、列印 PDF。
4. 建立 `README.md`：記錄功能、執行方式與設計取捨。
5. 準備部署到 GitHub Pages。

## 我如何判斷 AI 產出的東西可用

我沒有只看 AI 產出的文字，而是用以下方式檢查：

- 確認所有必要檔案存在。
- 用本機 HTTP server 開啟靜態網站。
- 檢查首頁 HTTP 回應是否成功。
- 檢查 HTML 是否包含主要欄位與功能文字。
- 檢查 JavaScript 沒有明顯語法錯誤。
- 部署後用公開網址讀回確認。

## 反思

這題真正考的不是「能不能做出最完整的報價系統」，而是在時間壓力下判斷什麼是可交付版本。我選擇先做靜態 Web 工具，原因是它最能降低部署與交付風險，也能讓主管立刻看到核心流程：輸入客戶與產品、計算金額、預覽報價單、列印 PDF。

如果有更多時間，我會把它擴充為正式內部系統，例如加入登入權限、雲端資料庫、報價單版本控管、主管簽核、寄信紀錄與正式 PDF 樣板。但在 1 小時實測中，我認為先交出穩定、可操作、可被驗證的版本，比做一個功能很多但不能準時交付的版本更重要。
